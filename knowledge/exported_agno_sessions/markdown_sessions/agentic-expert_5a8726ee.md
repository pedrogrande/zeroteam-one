### PROCESS CHARACTERISTICS (how it feels)

**P1. GUIDED, NOT BLANK**

**P2. BOUNDARIES FIRST**

**P3. PROGRESSIVE DISCLOSURE**
Start simple, add complexity as needed. One clear next step always. The user never faces all 18 framework questions at once. Each layer unfolds only when the previous one is complete. (ADHD principle: One Clear Next Step Always.)

**P4. EPISTEMICALLY HONEST**
The process shows confidence levels, flags assumptions, and distinguishes between what the user knows and what they're assuming about their own agent design. Every design decision carries epistemic metadata — not just in the output, but *during the process itself*. "You've defined this boundary with high confidence — would you like to challenge it?"

**P5. ENRICHING**
The process leaves the user more capable, not just with a deployed agent. The user understands their own workflow better, knows where boundaries should be, and can articulate why. The process is itself a learning experience. (H2: Amplification, Not Dependency — the user becomes a better agent designer, not a dependent one.)

**P6. RECOVERABLE**
If the process is interrupted, the user picks up where they left off — no lost work. If a wrong choice is made, the user can recover without starting over. Every artefact is persistent and every decision is revisitable. (ADHD principle: Replanning as Normal Workflow.)

---

### OUTPUT CHARACTERISTICS (what gets produced)

**O1. CONTRACT-NATIVE**
The agent's specification IS its enforcement contract. Design artefact and enforcement mechanism are unified — not a spec that then needs to be translated into enforcement. Every authority boundary is a guard. Every output has schema validation. Every budget has a halt condition. (Symmetric ternary: the contract is the contract, same for all actors.)

**O2. TYPED THROUGHOUT**
Every piece of information in the agent's design has an explicit type. Purpose is different from boundary is different from verification is different from cost budget. Types determine behavior — what's validated, what's enforced, what's displayed, what's connected to what.

**O3. DESIGN-TO-CODE TRACEABLE**
Every line of generated Agno code traces back to a specific design decision, and every design decision traces forward to its implementation. Bidirectional traceability is structural, not documented. Change a design decision → see which code changes. Change a code line → see which design decision it serves.

**O4. IDENTITY-PRESERVING**
The agent's identity (principal, purpose, boundaries, authority) is established early and preserved through the entire lifecycle — from design through deployment through operation through aging. The process never loses sight of what makes this agent THIS agent. (MVA Layer 0: Principal + Directive are established first and never overwritten.)

**O5. CONTEXT-AWARE**
The agent knows its context — its ecosystem, its relationships with other agents, its owner's domain, the trust environment it operates in. It's not an isolated creation; it's a member of a system from birth. (Composable: designed to work with others, not just alone.)

**O6. VERIFICATION-BUILT-IN**
The agent carries its own verification protocol. Not "verify after deployment" — the verification is designed into the agent's structure. Structural verification (schema), semantic verification (accuracy), authority verification (who decides) are all specified during creation, not bolted on afterward. (CC-1: Verification independence is designed, not assumed.)

**Agent's role:** Purpose Mirror (from your prior roster). Surfaces patterns, challenges assumptions, asks "what about...?" — but never decides what the target state IS.

### A5 — Subtask Goal & Risk Definition (Gap = 6)

**Why human-primary:** Risk imagination — "what could go wrong that nobody's thought of?" — is inherently creative and domain-experiential.
