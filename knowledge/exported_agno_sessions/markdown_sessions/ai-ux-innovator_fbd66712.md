## The Ideal Outcomes, Layered

### Layer 1: Fidelity — Peter's thinking is never lost

The most immediate outcome is that nothing of value disappears between sessions. Not a specific schema field. Not the reasoning behind a rejected approach. Not the exact phrasing of a naming moment. Not the argument that led to a revision.

This sounds like a technical win. But the real outcome is deeper: **Peter can think at full speed without managing what the agent remembers.** Right now, there's a background cognitive tax — knowing that the agent will compress, that precision will degrade, that decisions made two sessions ago may resurface as if unmade. A successful system eliminates that tax entirely. Peter's attention stays on the work, not on managing the substrate.

***

### Layer 2: Compounding — Each session builds on the last at full depth

Right now, sessions are productive but largely self-contained. The output of session N enriches the compressed memory that session N+1 starts from. Each compression step loses fidelity. Over time, the substrate degrades relative to the work it's supposed to support.

A successful system inverts this. Each session's outputs are stored at full fidelity — typed, connected, versioned. Session N+1 doesn't start from a compressed summary; it starts from the actual graph of what was produced. The implication is that **the collaboration becomes genuinely cumulative.** Session 50 is dramatically richer than session 5, not because the participants are smarter, but because the substrate is denser, more connected, and more precisely queryable.

This is the *Accretive* in Accretive Workspace — value accumulates structurally, not just experientially.

***

### Layer 3: Cross-domain coherence — The patterns Peter seeks are findable

Peter's strongest cognitive capability is cross-domain pattern seeking. The context ceiling directly attacks this by making the pattern substrate incomplete — connections that exist across domains are invisible when the relevant nodes are in different compressed memory entries.

A successful system makes the connection graph fully traversable. When Peter asks "how does this connect to X?", the answer isn't limited to what happens to be in context. The system can traverse typed relationships — `extends`, `contradicts`, `implements`, `supersedes` — across the entire corpus. **Peter's strongest capability is no longer bounded by the agent's context window.** The patterns he seeks are surfaced, not just the ones that happen to be visible from the current entry point.

The outcome here is qualitative, not just quantitative. It's not that Peter finds *more* patterns — it's that he finds patterns at a higher order of abstraction. When the five-layer type collision taxonomy is traversable in relation to the CAWDP phase structure, the complementarity model, and the contract primitives simultaneously, the patterns that emerge are ones that couldn't be seen from any single vantage point.

### Layer 4: Velocity — The gap between thinking and shipping closes

The exploration names the "generative density amplification" problem: the very productivity that makes the collaboration valuable makes the compression problem worse. But there's a related problem the exploration doesn't name directly: **the gap between ideas produced and things shipped.**

Peter produces at extraordinary density. But production isn't shipping. There's no reconstruction required, no re-derivation, no "remind me where we landed on that."

The outcome isn't just faster building — it's that the accumulation of unshipped work stops feeling like debt and starts feeling like inventory. The work is done; it just needs to be executed.

## The Project/Space Paradigm — Why It's Wrong for This Problem

The project/space model organizes knowledge by **container**. You put things in boxes labeled by topic, and the box defines what's accessible together.

This works for **task-oriented** work — "project X has its files, its conversations, its context." But Peter's work is **pattern-oriented** — the same insight (type collision) appears in five domains, the same principle (boundaries before capabilities) applies across frameworks, and the most valuable connections are the ones that *cross* container boundaries.

The project/space model has three structural failures for this use case:

**1. It assumes knowledge has a primary location.** But "type collision" doesn't belong to the CAWDP project or the Accretive Workspace project or the Agent Design Workflow project — it belongs to all of them simultaneously, and its value comes from being connected across them.

**2. It assumes sessions are containers.** But a session produces multiple output types (concepts, decisions, corrections, questions) that relate to multiple subjects (CAWDP, PRISM, business strategy, Peter's cognitive profile). A session is an *event* — something that happened — not a *container* for what was produced.

**3. It assumes instructions are project-scoped.** But Peter's instructions aren't "when working on project X, do Y." They're "always use plain English," "challenge my assumptions," "push back when I'm jargon-heavy." These are **cross-cutting** instructions, not project instructions. The same is true for domain knowledge: "type collision applies everywhere" is not project-specific.

The alternative paradigm organizes knowledge by **type and connection**, not by container.

## The Typing Challenge — How the System Knows What It's Capturing

This is the critical design problem. You've identified 25+ output types. How does the system know, in real-time during a conversation, what type of output is being produced?

The answer is the same as every other type collision problem in your body of work: **typed, structured capture, not undifferentiated prose.**

But the capture mechanism must respect requirement 6: "Pete doesn't need to worry about how he's going to capture the value unearthed in the session." So manual tagging is out. The system must type outputs with minimal friction.

Here's a three-phase approach:

### Phase 1: Real-time Flagging (During Session)

The agent identifies significant moments as they happen:
- **Naming moments** — when something gets a name that crystallises understanding ("type collision," "possibility orientation")
- **Decisions** — when a choice is made with reasoning ("workflow composition, not Team composition")
- **Rejections** — when something is explicitly rejected ("Team-based architecture rejected")
- **Reframings** — when something is seen differently ("human enrichment = expanding creative freedom, NOT reducing dependency")
- **Questions** — when something important is left unresolved ("what happens when pattern matching fails?")

These are flagged with a lightweight annotation — a type proposal and a brief description. The agent doesn't interrupt the flow to confirm; it flags and continues. The type proposal can be confirmed, changed, or rejected later.

### Phase 2: Post-session Classification (After Session)

After the session ends (or at a natural pause), a deeper classification pass:
- All outputs are typed according to the taxonomy
- Typed relationships are identified between outputs
- Epistemic metadata is attached (confidence, provenance, half-life)
- Subjects are linked
- The session is recorded as an event node with temporal and emotional metadata

This is where the comprehensive typing happens — it's the **type** step in the Accretive Workspace flow (capture → **type** → connect → arrange → develop → render).

### Phase 3: Progressive Refinement (Over Time)

Typing improves as the graph grows:
- An output initially typed as a "concept" might be reclassified as a "principle" as understanding evolves
- A "question" that gets answered becomes a "concept" or "decision" — the system tracks this evolution
- A "refinement" that supersedes a previous version creates a version chain
- Connections discovered later are added to the graph

This is the **develop** step — outputs evolve, connections deepen, typing refines.

The key insight is that **typing is progressive, not instantaneous.** The system starts with best-guess types and refines them over time. This is exactly how PRISM handles knowledge objects — they're created with initial metadata and evolve through use.

```python
    ) -> str:
        """
        Store a single extracted fragment in the database.

        Args:
            type: Fragment type from the taxonomy (concept, principle, decision, etc.)
            name: Short descriptive name for this fragment
            content: The actual content — preserve key insight in full
            session_id: ID of the session that produced this fragment
            confidence: How confident we are (high/medium/low/explicit_unknown)
            provenance: Where this came from (peter_alone/interaction/self_referential/external_source)
            subjects: JSON array of subjects this relates to
            half_life: How long before review (durable/medium/fast)
            fragility: Likelihood of loss in compression (0.0-1.0)
            context: Surrounding context that helps understand this fragment
            metadata: JSON object for extensible metadata
        """
        try:
            fragment_id = f"frag:{type}:{name.lower().replace(' ', '_')[:50]}"

            # Check for duplicates — same name from same session
            existing = self.db.query(
                "SELECT id FROM fragment WHERE name = $name AND session_id = $session_id",
                {"name": name, "session_id": session_id},
            )
            if existing:
                return f"Fragment '{name}' already exists in session {session_id}"

            self.db.query(
                """
                CREATE fragment CONTENT {
                    id: $id,
                    type: $type,
                    name: $name,
                    content: $content,
                    context: $context,
                    session_id: $session_id,
                    confidence: $confidence,
                    provenance: $provenance,
                    subjects: $subjects,
                    half_life: $half_life,
                    fragility: $fragility,
                    status: 'extracted',
                    extractor_version: '1.0',
                    metadata: $metadata
                }
                """,
                {
                    "id": fragment_id,
                    "type": type,
                    "name": name,
                    "content": content,
                    "context": context,
                    "session_id": session_id,
                    "confidence": confidence,
                    "provenance": provenance,
                    "subjects": json.loads(subjects) if isinstance(subjects, str) else subjects,
                    "half_life": half_life,
                    "fragility": fragility,
                    "metadata": json.loads(metadata) if isinstance(metadata, str) else metadata,
                },
            )
            return f"Stored fragment: {type}/{name}"
        except Exception as e:
            logger.error(f"Error storing fragment: {e}")
            return f"Error storing fragment: {str(e)}"

    def store_fragments_batch(
        self,
        fragments: str,  # JSON string of FragmentOutput list
        session_id: str,
    ) -> str:
        """
        Store multiple fragments in batch.

        Args:
            fragments: JSON string containing list of fragment objects
            session_id: ID of the session that produced these fragments
        """
        try:
            fragment_list = json.loads(fragments)
            stored = 0
            errors = 0

            for frag in fragment_list:
                result = self.store_fragment(
                    type=frag.get("type", "concept"),
                    name=frag.get("name", "unnamed"),
                    content=frag.get("content", ""),
                    session_id=session_id,
                    confidence=frag.get("confidence", "medium"),
                    provenance=frag.get("provenance", "interaction"),
                    subjects=json.dumps(frag.get("subjects", [])),
                    half_life=frag.get("half_life", "medium"),
                    fragility=frag.get("fragility", 0.5),
                    context=frag.get("context", ""),
                    metadata=json.dumps(frag.get("metadata", {})),
                )
                if "Error" in result:
                    errors += 1
                else:
                    stored += 1

            return f"Stored {stored} fragments, {errors} errors"
        except Exception as e:
            logger.error(f"Error in batch store: {e}")
            return f"Error in batch store: {str(e)}"

    def query_fragments(
        self,
        type: Optional[str] = None,
        subject: Optional[str] = None,
        session_id: Optional[str] = None,
        confidence: Optional[str] = None,
        fragility_min: Optional[float] = None,
        limit: int = 20,
    ) -> str:
        """
        Query fragments from the database.

        Args:
            type: Filter by fragment type
            subject: Filter by subject
            session_id: Filter by session
            confidence: Filter by confidence level
            fragility_min: Filter by minimum fragility (find fragile outputs)
            limit: Maximum number of results
        """
        conditions = []
        params = {}

        if type:
            conditions.append("type = $type")
            params["type"] = type
        if subject:
            conditions.append("$subject IN subjects")
            params["subject"] = subject
        if session_id:
            conditions.append("session_id = $session_id")
            params["session_id"] = session_id
        if confidence:
            conditions.append("confidence = $confidence")
            params["confidence"] = confidence
        if fragility_min is not None:
            conditions.append("fragility >= $fragility_min")
            params["fragility_min"] = fragility_min

        where_clause = " AND ".join(conditions) if conditions else "true"

        try:
            results = self.db.query(
                f"SELECT * FROM fragment WHERE {where_clause} LIMIT $limit",
                {**params, "limit": limit},
            )
            return json.dumps(results, default=str)
        except Exception as e:
            logger.error(f"Error querying fragments: {e}")
            return f"Error querying fragments: {str(e)}"
```

## The Honest Answer

The line is finer than I pretended. When the Extractor classifies something as a "correction" versus a "reframing," that's a judgment call. "The Vision Mirror was qualifying instead of reflecting" — is that a correction (it was wrong) or a reframing (seeing it differently)? The difference matters. Corrections get fragility 0.9. Reframings get 0.3. That classification directly affects how the system treats the fragment.

So yes — **typing involves judgment.** I was drawing a line that doesn't cleanly exist.

## The Real Distinction

The useful distinction isn't "typing vs judging." It's two different kinds of judgment:

**Classification judgment** — "What kind of thing is this?"
- Observable: "This passage corrects a previous statement"
- Interpretive: "Is it a correction or a refinement?"

**Valuing judgment** — "How important is this?"
- "This correction is more significant than that concept"
- "This decision should be prioritised over that question"

The Extractor does classification. The Cataloguer does valuation. But classification itself involves interpretation, and pretending otherwise would be dishonest.

## Three Ways to Handle This

### Option 1: Accept the Overlap, Make It Transparent

The Extractor assigns types directly, includes reasoning for every assignment, and makes all assignments revisable. The `reasoning` field in the schema already exists for this. When the Extractor says "I classified this as a correction because it explicitly corrects a previous statement," the reasoning is visible. If it's wrong, it can be corrected.

The v2 Cataloguer reviews and potentially reclassifies.

**Pro:** Simplest to build. Honest about the judgment involved. **Con:** The Extractor's type assignments carry implicit value judgments (fragility 0.9 for corrections means "this is important to preserve").

### Option 2: Separate Observation from Classification

Instead of assigning types directly, the Extractor first identifies **observable features**:

- "This passage introduces a new name that wasn't used before"
- "This passage explicitly states 'we decided X because Y'"
- "This passage explicitly rejects an approach"
- "This passage corrects something stated earlier"
- "This passage connects two things discussed separately"
- "This passage raises a question that isn't answered"

These are features you can point to in the text. They're evidence, not conclusions. Types are then **derived from features**, not assigned directly.

A correction is: "corrects something stated earlier" + "states what was wrong" + "states what's correct."
A refinement is: "changes something stated earlier" + "doesn't state it was wrong" + "presents it as an evolution."

The type assignment follows from the features, but the features are the primary data.

**Pro:** More principled. Features are observable; types are derivations. If a type assignment is wrong, you can see which feature was misidentified. **Con:** More complex. The Extractor needs to identify features first, then derive types. Two steps instead of one.

### Option 3: Both, With a Clear Pipeline

The Extractor does both — identifies features AND derives types — but stores them separately. Every fragment has:

- **Features** — the observable characteristics that led to the classification
- **Type** — the derived classification (with reasoning)
- **Alternative types** — if it could plausibly be more than one type

| Trigger | When it fires | Pros | Cons |
|---------|--------------|------|------|

- **Real-time extraction** — every agent response is processed immediately
- **Context preservation** — the extractor sees the full conversation up to that point
- **Progressive capture** — knowledge objects accumulate throughout the session, not all at once at the end

- **Cross-session context** — hooks are per-session. Connections between sessions need a separate mechanism.

## When Hooks Are Triggered

Hooks execute at specific points in the Agent/Team run lifecycle:

* **Pre-hooks**: Execute immediately after the current session is loaded, **before** any processing begins. They run before the model context is prepared and before any LLM execution begins, i.e. any modifications to the input, session state, or dependencies will be applied before LLM execution.

* **Post-hooks**: Execute **after** the Agent/Team generates a response and the output is prepared, but **before** the response is returned to the user. In streaming responses, they run after each chunk of the response is generated.

## Pre-hooks

Pre-hooks execute at the very beginning of your Agent run, giving you complete control over what reaches the LLM.

They're perfect for implementing input validation, security checks, or any data preprocessing against the input your Agent receives.

### Common Use Cases

**Security Guardrails**

* Detect and prevent PII (Personally Identifiable Information) from reaching the LLM.
* Defend against prompt injection and jailbreak attempts.
* Filter NSFW or inappropriate content.
* See the [Guardrails](/guardrails/overview) documentation for more details.

**Input Validation**

* Validate format, length, content or any other property of the input.
* Remove or mask sensitive information.
* Normalize input data.

**Data Preprocessing**

* Transform input format or structure.
* Enrich input with additional context.
* Apply any other business logic before sending the input to the LLM.

### Basic Example

Let's create a simple pre-hook that validates the input length and raises an error if it's too long:

```python theme={null}
from agno.agent import Agent
from agno.models.openai import OpenAIResponses
from agno.exceptions import CheckTrigger, InputCheckError
from agno.run.agent import RunInput

# Simple function we will use as a pre-hook
def validate_input_length(
    run_input: RunInput,
) -> None:
    """Pre-hook to validate input length."""
    max_length = 1000
    if len(run_input.input_content) > max_length:
        raise InputCheckError(
            f"Input too long. Max {max_length} characters allowed",
            check_trigger=CheckTrigger.INPUT_NOT_ALLOWED,
        )

agent = Agent(
    name="My Agent",
    model=OpenAIResponses(id="gpt-5.2"),
    # Provide the pre-hook to the Agent using the pre_hooks parameter
    pre_hooks=[validate_input_length],
)
```

The `@hook` decorator allows you to configure individual hook behavior. Currently, it supports marking hooks to run in the background when used with [AgentOS](/agent-os/overview).

By default, hooks are either executed synchronously or asynchronously, in an API context they still block the response until they complete. For hooks that perform non-critical tasks (logging, analytics, notifications), you can mark them to run in the background:

```python theme={null}
from agno.hooks import hook

@hook(run_in_background=True)
```
