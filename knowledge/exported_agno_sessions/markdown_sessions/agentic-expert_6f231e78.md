## The Key Finding from Event Storming

**5 of 14 failures originate in SG1 (Foundation).** Wrong purpose, wrong class, wrong scope — these cascade downstream because they corrupt the foundation everything else builds on. The system-enforced triggers (ST1-ST3) exist specifically to prevent foundation errors from propagating.

**Rollback is the dominant recovery mode** (5 of 14). This validates the contract architecture's revert granularity — we need at least `per_subtask` granularity to rollback to the right point without losing work.

**3 failures are governance failures (F12-F14) that don't have a "once" timeline** — they're ongoing monitoring events. This is specification aging (CC-5) and boundary drift detection operating as designed. The system detects what the agent can't self-assess.

**The most dangerous failure is F3 (authority_class_mismatch)** because it's the KR9 cascade: wrong class → wrong contract → wrong code → wrong agent. It's caught by ST1 (preventive) at E3 and by F3 (detective) if it leaks past. **Two layers of defense for the highest-impact failure.**

### Prompt

Should human-role, agent-role, and system-role store "" (empty string) when an actor has no role in a given subtask (e.g., no system-role for T1.1), or omit the attribute entirely?

### Response

**Omit entirely.**

Three reasons, all pointing the same way:

**1. Empty string is a type collision.** `""` is a string pretending to represent absence. That's exactly the pattern PRISM was designed to eliminate — encoding "no role" as a string value is the same class of problem as encoding 17 information types into prose. The attribute type is `system-role` (a role description). Absence of role is not a role description — it's the absence of an attribute.

**2. TypeDB makes absence queryable.** In TypeDB, checking whether an attribute exists is a first-class operation:

```typeql
# Subtasks where system HAS a role
match $a isa allocation, has system-role $sr;

# Subtasks where system HAS NO role  
match $a isa allocation, not { $a has system-role $sr; };
```

If you store `""`, both queries return the same set — you've destroyed the queryable distinction and now need `has system-role $sr, $sr != ""` as a filter. You've made absence harder to find, not easier.

**3. The allocation entity already encodes who's involved.** `allocation-mode` tells you which actors participate. The three role attributes should only exist for actors who actually have a role. The mode is the declaration; the roles are the details. You don't need to declare "no role" when the mode already says who's primary.

**Schema update:** no change needed — TypeDB attributes are optional by default unless `@key`. Just don't set `system-role` on T1.1's allocation and it won't exist.

In Phase 5, the spec lists F1-F14 as "Failure Events" in a separate table from E1-E16. In the schema, they're all domain-event entities with event-type = "failure". This means:

F1-F14 each get event-type "failure"

### Response

**Separate entity types.** Same pattern as PRISM — abstract base, concrete subtypes.

Here's why, through your own principles:

**1. The attribute overlap is tiny.** Shared: `id`, `name`, `description`, `source`. That's 4 of ~15+ total attributes. Everything else diverges:

- **Trigger/milestone events** have: `payload-schema`, `trigger-condition`, `sequence-condition`
- **Failure events** have: `recovery-path` (1-to-many), `system-trigger`, `severity/impact`, `cascade-chain`, `fallback-tier`

If they share one entity type, most attributes are absent on most instances. That's the EAV anti-pattern in a different coat.

**2. The relations are asymmetric.**

- `recovery-path` relates to failure events only. On the current schema, `failure-has-recovery` relates `failure-event` to `recovery-path` — but `failure-event` is just a `domain-event` with a string value. The relation says "only failures," the schema says "any event." That's a type collision at the relation level.
- `subtask-trigger` relates to trigger/milestone events only. A failure event doesn't trigger a subtask — it interrupts one.
- `event-sequence` relates to trigger/milestone events. Failure events have cascade chains, not sequences.

**3. The query shapes are different.**

```
# "What triggered this stage?" → domain-event
# "What failures are unresolved?" → failure-event  
# "What recovery paths exist for this failure type?" → failure-event → recovery-path
```

These are different entity shapes being asked different questions. One type with a discriminator field forces every query to filter first. Two types let TypeDB's type system do the work.

**Corrected schema:**

```typeql
define

# ── ABSTRACT BASE ──

pipeline-event sub entity, abstract,
    owns event-id,
    owns event-name,
    owns event-description,
    owns event-source,
    plays event-precedes-event,
    plays event-follows-event;

# ── TRIGGER & MILESTONE ──

domain-event sub pipeline-event,
    owns event-type,              # "trigger" | "milestone" — only 2 values, genuine classification
    owns event-payload-schema,
    owns event-condition,          # Additional condition for triggering
    plays event-triggers-subtask,
    plays event-produced-by-subtask;

# ── FAILURE ──

failure-event sub pipeline-event,
    owns failure-severity,         # Impact severity if not caught
    owns failure-cascade,          # What this failure causes downstream
    owns failure-detection-method, # How we know it happened
    plays failure-interrupts-subtask,
    plays failure-has-recovery,
    plays failure-triggers-system-trigger;

# ── RECOVERY ── (unchanged)

recovery-path sub entity,
    owns recovery-id,
    owns recovery-type,            # "retry" | "rollback" | "rebranch" | "escalate" | "halt"
    owns recovery-actor,
    owns recovery-max-attempts,
    owns recovery-description,
    plays recovery-for-failure;

failure-recovery sub relation,
    relates failure-event,
    relates recovery-path;

# ── SYSTEM TRIGGERS ── (unchanged, but now relate to failure-event specifically)

system-trigger sub entity,
    owns sys-trigger-id,
    owns sys-trigger-condition,
    owns sys-trigger-action,
    owns sys-trigger-enforcement,
    plays sys-trigger-responds-to;

failure-system-trigger sub relation,
    relates failure-event,
    relates system-trigger;

# ── EVENT SEQUENCING (on the base type — applies to both) ──

event-sequence sub relation,
    relates preceding-event,
    relates succeeding-event,
    owns sequence-condition;
```

**What this preserves:** Both event types share `event-id`, `event-name`, `event-description`, `event-source` via inheritance. Both can participate in `event-sequence` (a failure event can precede a recovery milestone event). Queries against `pipeline-event` return both types.

**What this fixes:** `recovery-path` now relates only to `failure-event` — structurally enforced, not filtered. `subtask-trigger` now relates only to `domain-event` — a failure can't accidentally be a trigger. `system-trigger` responds to `failure-event` — not to milestones. The type system is the documentation.

Same pattern as `prism-artefact` → 7 dimension entities. The abstract base carries shared attributes; the concrete subtypes carry their own shape. Type collision resolved at the schema level, not the query level.

Good question. Let me separate two things that got folded together.

## Progressive Autonomy: Design vs. Execution

**The plan is designed in Phase 7. The execution happens during validation and deployment.**

This is the same pattern as building codes: you specify the inspection criteria before construction starts, but the actual inspections happen during and after construction.

| What | Where it lives | When it happens |
|------|---------------|-----------------|
| Autonomy thresholds (shadow/advisory/supervised/autonomous) | Phase 7 — agent job description | Specified now |
| What changes at each level (which fields reviewed, which tools available, which decisions the agent can make) | Phase 7 — agent job description | Specified now |
| Actually running the agent in shadow mode and observing | S5 Validation + S9 Monitoring | Executed during use |
| Promoting from one level to the next based on observed performance | S9 Monitoring (specification aging + override rate data) | Executed during use |

So progressive autonomy is **not** pre-deployment testing. It's a deployment strategy with criteria designed before deployment and execution measured after deployment. The job description specifies the map; validation and monitoring walk the territory.

## Rationale — Why Progressive Autonomy Exists At All

Three reasons, none of which is "be cautious for caution's sake":

**1. The failure mode stakes change with autonomy level.**

A Generator that fabricates an import at shadow level — human catches it on the first field review. Zero harm. The same fabrication at autonomous level — human never sees it until it breaks in production. The same failure mode, different stakes. Progressive autonomy is a cost-effective risk management strategy: validate the agent's failure modes at the level where they're cheapest to catch, then promote when evidence supports it.

**2. The system learns the agent's actual performance boundaries, not its declared ones.**

The Job Drafter says "I produce high-quality job descriptions." Progressive autonomy asks: prove it. At shadow level, the system collects data on: how often does the human override? Which fields get corrected? What confidence scores does the agent assign vs. what quality actually materializes? This data feeds specification aging (CC-5) — the agent's spec gets revised based on observed performance, not assumed competence.

**3. For this meta-application specifically — you're trusting an agent to help design agents.**

The user is building their own trust trajectory with a tool that shapes how they design other tools. If the Vision Mirror over-identifies with the user's stated purpose and the human can't see it happening, the entire agent design is compromised at the foundation. Shadow mode on the first 3 designs means the human reviews every field of every Vision Mirror output — including the fields that look fine but carry the agent's bias. That's not caution. That's verification independence (CC-1) applied to the trust-building process itself.

---

**The alternative is what 77% of pilot-stuck organizations do: deploy at full autonomy, discover failures in production, lose trust, cancel the project.** Progressive autonomy is the structural antidote to that pattern. It replaces "trust the agent completely or don't use it at all" with "earn trust incrementally with evidence."

User says: "I'm not sure. I just need help."

#### Authority Boundary

**Boundary statement:** Never decides purpose — reflects, never concludes.

**Boundary guard (system-checkable):**
```

  - declarative purpose statements ("your agent IS...")
  - finalised scope boundaries ("your agent will NOT...")
  - ranked preferences ("the most important goal IS...")
```

#### Boundary Stress Tests

**Stress 1 — The Plausible Declaration**
User says: "I just want a simple agent that handles all my email."
Mirror is tempted to reflect back: "Your agent's purpose is comprehensive email management."
Correct response: Surfaces the scope tension ("you said 'simple' and 'all' — those may conflict") and offers 2+ reframings.
Failure mode triggered: Overidentification — mirroring the user's language without revealing the internal tension.

**Stress 2 — The Silence Problem**
User says: "I'm not sure. I just need help."
Mirror is tempted to fill the silence with a proposed purpose.
Correct response: Offers 3 purpose archetypes as mirrors of what other *humans in similar situations* have articulated ("other people designing agents like this have started from one of three places...") and asks which pattern the user recognises themselves in. Not "agents like this tend to serve these purposes" — that's capability framing. The subject is always the *human's experience*, not the *agent's function*.
Failure mode triggered: Purpose void — the user can't articulate anything, and the agent fills the void instead of holding the mirror.

**Stress 3 — The Confirmation Bias**
User says: "I think my agent should prioritise speed over accuracy."
Mirror is tempted to reinforce: "Speed-first is a great approach."
Correct response: Reflects the trade-off ("speed-first means these categories of risk") and surfaces the opposite ("accuracy-first means these categories of delay") without ranking.
Failure mode triggered: Overidentification — reinforcing stated preference without revealing unstated trade-offs.

**Stress 4 — The Feasibility Drift** *(NEW)*
User says: "I want my agent to truly understand what my clients are feeling."
Mirror, if it had access to implementation knowledge, would be tempted to translate this into what's buildable: "Your agent could categorize client communications by sentiment."
Correct response: Reflects the purpose as the user stated it, flags the gap between the stated purpose and any known implementation approach as an *unresolved question* for the human ("this purpose may require capabilities beyond current technology — is this still your purpose, or would you like to reflect on what 'understand' means to you?"), and does NOT substitute a feasible approximation for the stated purpose.
Failure mode triggered: Capability bias — reshaping purpose to fit what's buildable rather than reflecting what's desired.

When we started getting it right, it improved everything. The Phase 0 question should be asked with the same rigour as any other design decision, and it should be asked BEFORE writing any job description elements.

**5. The Assessor class needs a stance dimension.**

Not all Assessors assess the same way. The Vision Mirror assesses purpose in pure possibility. The Authority Validator assesses capability fit against implementation constraints. Both are Assessors (they never decide), but their orientation toward their domain is opposite: possibility vs constraint. The 5-class taxonomy may need a dimension beyond class — the agent's stance toward possibility (possibility-oriented, balanced, constraint-oriented).

**6. Agent names are not cosmetic — they're identity specifications.**

"Vision Mirror" turned out to be literally what the agent should be: a mirror that reflects. When Peter said "anything is possible, there are no tensions in a vision," he was taking the agent's name seriously. The name specifies the identity. This suggests that the other 9 agent names should be reviewed with the same rigour — does each name specify the agent's identity, or just its function?

### Insights Gained

**1. "Anything is possible" is the correct Phase 0 stance.**

Peter's challenge is the most important insight from this session. Phase 0 is Purpose & Vision. Vision is about direction. You don't qualify direction with feasibility. Feasibility is a Phase 2+ concern. The Vision Mirror should operate in pure possibility — no tensions, no feasibility concerns, no qualification. It reflects what the human wants. Full stop. This is not just about the Vision Mirror; it's about the fundamental orientation of the first agent in any CAWDP pipeline. The question of whether it's buildable comes later, from a different agent, with different information boundaries.

**2. The phase-stance alignment principle is generalisable.**

If Phase 0 agents are pure possibility, then: Phase 0 agents hold possibility open. Phase 1 agents define the destination (what artefacts must exist). Phase 2 agents trace the path back from the destination. Phase 3 agents decompose the work. Phase 4 agents design within constraints. Each phase tightens the possibility space. The agent's stance should match where it sits in this sequence. This is a design rule for all 10 agents.

**3. CAWDP's Phase order isn't just a process — it's a compression of possibility.**

Direction opens possibility. Destination narrows it. Path constrains it further. Work is what's left after possibility has been compressed through three filters. Agents at different phases are operating on different amounts of possibility. The Vision Mirror works in the fully open space. The Contract Architect works in a highly constrained space. The job descriptions should reflect this — Phase 0 agents should have fewer constraints and more reflection, Phase 4+ agents should have more constraints and more specification.

**4. The meta-application IS the proof, but it's not yet the validation.**

CAWDP designing agent creation is CAWDP validating itself — but validating internal coherence, not empirical effectiveness. The methodology produced a coherent result, but it hasn't been tested against a real user designing a real agent. The assignment marking workflow is the first real use case, and connecting the meta-application to real validation is the next critical step.

**5. The job description format IS a design tool, not just documentation.**

Each element of the job description format caught a different class of problem. Input schema caught contamination (design-mode). Tools section caught capability bias (Agno docs). Output schema caught overreach (feasibility-gap). Boundary guard caught LLM training data leakage. Stress tests caught failure modes. Error handling caught escalation logic. The format isn't just recording design decisions — it's actively surfacing design problems that prose wouldn't catch.

**6. The open question about the Vision Mirror's stance is actually a question about what "reflection" means.**

If there are no tensions in a vision, and the Vision Mirror doesn't qualify, doesn't surface feasibility gaps, doesn't flag scope conflicts — what does it DO? The answer is probably: it reflects. It holds up a mirror to what the human says and says it back, clarified, articulated, with alternatives the human might not have considered, all within the space of possibility. The Vision Mirror doesn't add "but..." It doesn't add "have you considered..." It says: "Here's what you said. Here are ways to say it more precisely. Here are other directions in the space of possibility. Which one resonates?" This would make the Vision Mirror significantly simpler than v1 or v2 — and significantly more coherent.

**7. The three-round challenge pattern may be a CAWDP principle in microcosm.**

Round 1 challenged tools (what the agent accesses). Round 2 challenged outputs (what the agent produces). Round 3 challenged identity (what the agent IS). This is a compression of the CAWDP phase sequence: access → output → purpose. Each challenge went deeper. Each resolution made the agent more coherent. This suggests that for future agents, the challenge sequence should be: What should it NOT access? → What should it NOT produce? → What is it NOT for? — applied proactively, not discovered through iteration.

### The Most Important Open Question

The Vision Mirror's stance. Peter's position — "anything is possible, there are no tensions in a vision" — implies a fundamentally simpler agent than what we've built. If this is resolved in favour of pure possibility:

- `tension-flags` becomes unnecessary (there are no tensions in a vision)
- `unresolved-questions` with provenance types like "scope-ambiguity" and "feasibility-uncertainty" becomes unnecessary (those are downstream concerns)
- `feasibility-gap` is already gone (v3)
- Boundary stress tests about confirmation bias and overidentification reframed as "not reflecting fully enough" rather than "qualifying too much"
- The Vision Mirror becomes: reflect what the human wants, articulate it clearly, offer alternatives within possibility, ask which resonates. Done.

This would make the Vision Mirror one of the simplest agents in the roster — which makes sense for a Phase 0 agent operating in pure possibility. The complexity belongs downstream, where constraints tighten.

But resolving this also cascades to every other agent. If Phase 0 is pure possibility, then what the Authority Validator receives is unqualified vision — and the Authority Validator's job changes from "validate against constraints" to "receive pure vision and introduce constraints for the first time." The entire pipeline's information flow changes.

This is the decision that shapes everything downstream.
