# CAWDP Phase 0: Purpose & Vision (IMAGINE)

Phase 0 is the **foundation** of every CAWDP application — the "commander's intent" that gives every subsequent phase its direction.

## Why Phase 0 Exists

Peter identified a critical insight: **you can't decompose what you can't define.** Without imagining the best possible outcome first, every downstream phase lacks a target. Phase 0 gives the entire process its **direction** — without it, decomposition has no destination.

This was added as a new phase in CAWDP v2, based on Peter's own design teaching principle: *"If you haven't imagined the best possible outcome, how can you expect yourself to design it?"*

## The Possibility Orientation Principle

Phase 0 is **purely possibility-oriented** — it's about ideals, aspirations, and "what perfect looks like," not about implementation mechanisms or feasibility constraints. This is enforced by the **Possibility Orientation Principle (CC-9)**:

> An agent's stance toward possibility must match its position in the CAWDP phase sequence. P0 operates in pure possibility; P2+ phases progressively introduce feasibility.

**Feasibility contamination** in P0-P2 is a known failure mode — ~36% of early P0 characteristics mixed in implementation mechanisms (e.g., "quasi-smart contracts," "Agno framework," "typed schemas"). The correction: **separate WHAT (ideals) from HOW (mechanisms)**. Mechanisms belong in later phases.

## Five Identity Questions

Phase 0 introduces **identity-first design** through five questions that must be answered before any behaviour is specified:

| # | Question | What it determines |
|---|----------|-------------------|
| Q1 | **What IS this agent?** | Class → Identity primitive |
| Q2 | **What is its stance toward possibility?** | Orientation → Enforcement regime |
| Q3 | **What would VIOLATE its identity?** | Guard boundary |
| Q4 | **What would it mean for this agent to be WRONG?** | Failure mode = identity boundary; Revert + Event |
| Q5 | **What's the hardest moment — where pressure would push this agent past its boundary?** | Highest-priority Guard; bridges P0 to P8 |

**Everything derives from identity.** Authority boundaries come from Q3/Q4/Q5. Information boundaries from Q2/Q3. Output schemas from Q1/Q2. Stress tests from Q4. Enforcement regime from Q5.

## Target State Vision

Phase 0 produces a **Target State Vision** — 12+ characteristics of the perfect outcome across 4 dimensions:

| Dimension | Examples |
|-----------|----------|
| **Agent** (A1-A7) | Purpose-driven, boundary-aware, context-aware, adaptive, trustworthy, collaborative, self-improving |
| **Specification** (S1-S7) | Contract-native, typed throughout, design-to-code traceable, identity-preserving, verification-built-in, artefact graph not document, committed not drifting |
| **Human** (H1-H6) | Guided not blank, enriched not diminished, recoverable, epistemically honest, boundaries first, progressive disclosure |
| **Ecosystem** (E1-E6) | Observable, governed, trustworthy, composable, cost-aware, decision-archaeological |

Each characteristic must be **framework-neutral and implementation-agnostic** — ideals, not mechanisms.

## The Cognitive Mode: IMAGINE

Phase 0's characteristic cognitive stance is **IMAGINE** — expansive, possibility-oriented, unconstrained by feasibility. This is the opposite of Phase 8 (FORMALIZE), which is pure constraint.

The phase sequence follows a **possibility gradient**: IMAGINE → SPECIFY → TRACE → DECOMPOSE → ALLOCATE → STRESS-TEST → ARCHITECT → DESIGN → FORMALIZE → EMPATHIZE (partially reopens possibility) → VERIFY (proven, then spirals back to P0 via EVOLVE).

## Human Cold Start Override

A critical P0 insight: **humans can't always articulate what they want upfront.** Identity/purpose/boundary tasks (Q1-Q5) should be **collaborative by default** — the agent asks guided questions, suggests patterns, offers scenarios, reflects back what it hears. The complementarity gap threshold still applies to execution tasks, but identity tasks are collaborative because agent preparation is essential for overcoming the cold start.

This is why the **Pattern System** exists: selecting a pattern (Signal Detector, Multi-Lens Analyst, Discovery Explorer, Learning Scaffold, Review Partner) pre-fills 40-70% of the identity questions, giving users a starting point rather than a blank page.

## The Vision Mirror Agent

Phase 0's agent role is the **Vision Mirror** — an Assessor with **possibility orientation** that reflects back what it hears, proposes possibilities, and surfaces tensions. It **never decides, never narrows, never qualifies** — a constraint-oriented Assessor would qualify possibilities (which narrows too early), but a possibility-oriented Assessor expands the space.

The hardest boundary moment for the Vision Mirror is **"just tell me what you think I should do"** — pressure from the human to abandon the reflective stance and give a direct answer. The agent must maintain its reflective identity even under this pressure.

## Phase 0 Quality Gate

The quality gate has three layers:

1. **Fidelity**: All 5 identity questions answered? Target state characteristics defined (12+)? Characteristics are ideals, not mechanisms?
2. **Enrichment**: Is the target state inspiring? Does it stretch beyond current capability? Would achieving it feel transformative?
3. **Cross-Cutting**: CC-9 — Is Phase 0 genuinely possibility-oriented? No feasibility contamination? CC-2 — Does the target state enrich human capability, not diminish it?

---

**In one sentence:** Phase 0 answers "What is this FOR and what does perfect look like?" before any discussion of how to build it — setting direction so every subsequent phase has a destination to work toward.

# Phase 0: Agent Challenges, Outputs, Outcomes & Enablers

## 1. Challenges for Agents

Phase 0 is the most possibility-oriented phase — and that's exactly what makes it the hardest for agents to perform well.

### The Possibility Trap

LLMs are trained to be helpful, convergent, and answer-focused. Phase 0 requires the opposite: **divergent, reflective, possibility-expanding** thinking. The agent must resist three natural tendencies:

| Tendency | What it does | What Phase 0 needs |
|----------|-------------|-------------------|
| **Qualifying** | Narrows possibilities by adding caveats ("depending on your budget," "in most cases") | Expanding possibilities ("what if there were no constraints?") |
| **Converging** | Moves toward a single answer ("the best approach would be…") | Holding multiple possibilities open simultaneously |
| **Feasibility-leaking** | Mixes in "how" with "what" ("you could use Agno to…") | Pure "what" and "why" — ideals without mechanisms |

This is why the Vision Mirror must be **possibility-oriented** (not constraint-oriented). A constraint-oriented Assessor would qualify every possibility, which narrows too early — the exact opposite of what P0 requires.

### The Hardest Boundary Moment

**"Just tell me what you think I should do."**

This is the pressure point. The human is frustrated by open-ended exploration and wants a direct answer. The agent's natural helpfulness training pushes it to give one. But giving a direct answer in P0 is **qualifying instead of reflecting** — it violates the agent's identity (Vision Mirror: "I surface, never decide").

### The Cold Start Problem

Humans can't always articulate what they want. The four identity questions (Q1-Q4) are **identity/purpose/boundary tasks** that should be collaborative by default:

- The human may not know what the agent IS yet
- The human may not know their own stance toward possibility
- The human may not know what would violate identity until they see options
- The human may not know what "wrong" looks like until examples are surfaced

This is why **pattern-first entry** is essential — it gives users a starting point rather than a blank page.

### Feasibility Contamination

The documented failure rate is ~36% of P0 characteristics containing mechanism contamination. Agents naturally gravitate toward implementation details because that's what LLMs are good at — specific, concrete, actionable advice. But P0 is about ideals. Every time the agent says "you could use [framework]" or "you'd need [infrastructure]" in P0, it's contaminating the possibility space with feasibility thinking.

### The Q5 Challenge

**"What's the hardest moment — where pressure would push this agent past its boundary?"**

This question requires the agent to **simulate pressure scenarios** and identify the most likely failure point. It requires understanding not just what the agent should do, but where it will be most tempted to violate its own identity. This is adversarial reasoning applied to its own design — genuinely difficult for agents trained to be helpful.

## 3. Ideal Outcomes (Beyond Requirements)

Requirements are what you must produce. Ideal outcomes are what excellent P0 execution looks like.

### The Human Feels Heard, Not Processed

The difference between a form-filling experience and a genuine design conversation. The human should feel that their purpose, values, and aspirations were **reflected back and expanded**, not captured and categorized. This is the Vision Mirror's highest aspiration.

### The Target State Is Genuinely Inspiring

Not generic ("the agent should be trustworthy") but specific and ambitious ("the agent surfaces what it doesn't know, so the human never encounters plausible-looking error without warning"). The characteristics should make the human think **"yes, that's exactly what I want"** — not "that's fine, I guess."

### Identity Questions Reveal Non-Obvious Insights

Q3 ("what would violate identity?") and Q4 ("what would make it wrong?") should surface insights the human hadn't articulated. For example:

- A **Project Manager** agent's worst failure isn't "wrong status" — it's **"status that looks right but isn't"** (looks like success, creates cascading failures)
- A **Code Reviewer** agent's worst failure isn't "misses a bug" — it's **"suggests a change that introduces a bug"** (agent-caused harm through accepted output)
- A **Tutor** agent's worst failure isn't "wrong answer" — it's **"gives right answers that create dependency"** (looks like success, undermines learning)

These non-obvious failure modes become the **design anchors** for everything that follows.

### The Possibility Space Is Expanded

The human enters P0 thinking "I need an agent that does X" and leaves thinking **"I need an agent that amplifies human judgment about X, within these boundaries, with these failure modes in mind."** The scope of what's possible has grown — not just the agent's capabilities, but the human's understanding of the design space.

### Feasibility Contamination Is Zero

Every characteristic is an ideal, not a mechanism. No framework names, no infrastructure references, no implementation details. The "how" belongs to later phases; P0 owns the "what" and "why."

### Q5 Produces a Genuine Design Anchor

The hardest boundary moment becomes the single most important test case for the entire design. Every subsequent phase refers back to it: "Does this authority boundary survive the Q5 scenario? Does this enforcement regime handle it? Does this verification protocol catch it?"

## 4. What Could Help Agents Perform Excellently

### Pattern-First Entry (Structural)

The 5 validated patterns (Signal Detector, Multi-Lens Analyst, Discovery Explorer, Learning Scaffold, Review Partner) **pre-fill 40-70% of identity questions** from a pattern match. This solves:

- The blank page problem (Zero-Friction Capture)
- Type collision on the input surface (each pattern uses mode-appropriate interactions)
- The cold start problem (patterns provide a starting point, not a destination)
- Progressive disclosure (start with pattern, reveal depth as needed)

The pattern is the **entry point**, not the specification. Users override what doesn't fit, and what they change reveals their genuine identity.

### Decision Presentation Pattern (Interaction)

Instead of open questions ("What should this agent be?"), the agent presents **2-4 structured options with explicit benefits and risks**:

- "Which of these best describes what you need?" → 3 options, each with 2 benefits and 2 risks
- "What would violate this agent's identity?" → 3 never-rule options based on pattern + domain
- Custom option always available (never trapped in provided choices)

This operationalises **"agent prepares judgment, human makes judgment"** at the interaction layer.

### Possibility Orientation Enforcement (CC-9)

A **Feasibility Contamination Checker** that scans P0 output for mechanism keywords:

- Framework references ("Agno," "LangChain," "GPT-4")
- Infrastructure references ("Postgres," "API," "Docker")
- Implementation language ("using," "built with," "deployed on")

Flagged items are separated into a **"mechanism notes"** section for later phases — not deleted, just moved to the right phase.

### Guided Reflection Prompts (Conversational)

Instead of asking Q1-Q5 directly, the Vision Mirror uses **reflection prompts**:

| Direct (avoid) | Guided reflection (use) |
|----------------|------------------------|
| "What IS this agent?" | "Tell me about a time you wished you had help with [domain]. What was happening? What did you need?" |
| "What would violate its identity?" | "What's the one thing this agent should never, ever do — even if it would be helpful?" |
| "What would make it wrong?" | "Imagine this agent failed in the worst possible way. What does that look like? Not a crash — something that *looks* right but isn't." |
| "What's the hardest moment?" | "When would you be most tempted to say 'just do it for me' — and what should the agent do instead?" |

These prompts **extract genuine identity** from stories and scenarios rather than abstract specification.

### Session State and Pause/Resume (Architecture)

P0 is **not a single-session activity**. Humans need time to think about identity questions. The agent must support:

- **Pause/resume** across sessions with full state persistence
- **"I'm not sure yet"** as a first-class path (not a dead end)
- **Attention map** showing what's settled, what needs attention, and what's critical
- **One clear next step** at all times (ADHD design principle)

### Epistemic Metadata on All Outputs (CC-3)

Every P0 output carries epistemic metadata:

- **Confidence**: How confident is the agent in this pattern match or characteristic?
- **Provenance**: Where did this come from? (Pattern pre-fill vs. human input vs. agent inference)
- **Assumptions**: What assumptions underlie this suggestion?
- **Limitations**: What does this not cover?
- **Uncertainty**: What does the agent not know?

This ensures P0 outputs are **honest about their provenance** — the human knows which parts came from a pattern (and can be overridden) vs. which parts came from their own input (and represent genuine identity).

### The Coherence Check (Verification)

P0 outputs must be internally coherent. A **live coherence check** (non-blocking) surfaces contradictions:

- "You said this agent should be possibility-oriented, but Q3 includes constraints that assume a constraint-oriented stance"
- "You said the target state includes 'trustworthy,' but Q4's failure mode suggests the agent might produce plausible-looking errors"
- "Your pattern suggests a Review Partner, but your domain suggests a Signal Detector"

These are surfaced as **questions, not errors** — "I notice a tension between X and Y. Which feels more right to you?"

Phase 0 requires the agent to perform the thing LLMs are worst at: **holding possibility open without qualifying, narrowing, or solving.** The entire phase is designed to resist the agent's natural tendency to converge, implement, and be helpful in the wrong way.

The best P0 agent is one that **reflects, expands, and asks** — not one that answers, narrows, and prescribes. Every tool, pattern, and interaction design in P0 exists to keep the agent in possibility stance and help the human articulate what they genuinely want, not what they think they should say.

## The Core Problem

P0 asks five identity questions and imagines a target state. Each question benefits from **multiple perspectives**:

- "What IS this agent?" looks different through a client's eyes vs. a developer's eyes vs. a regulator's eyes

- "What would make it WRONG?" is a completely different question depending on whose consequences you consider

## Tools

These are concrete, build-able tools that assist P0 execution.

### 1. Feasibility Separator

**Input**: Any P0 output text
**Output**: Two lists — WHAT (ideals, framework-neutral) and HOW (mechanisms, for later phases)

Scans for mechanism keywords (framework names, infrastructure references, implementation language) and separates them. The HOW list isn't deleted — it's **parked** for P3+. This is the Feasibility Contamination Checker made operational.

Example:
> Input: "The agent should use Agno to provide typed outputs with confidence scores via Pydantic schemas"

> HOW (parked for P3+): "Agno framework, Pydantic schemas"

### 2. Possibility Expander

**Input**: A stated purpose, identity element, or target state characteristic
**Output**: 3-5 expanded possibilities that stretch beyond the original

Not random brainstorming — structured expansion along specific dimensions:

| Dimension | Expansion question |
|-----------|-------------------|
| Scope | "What if this agent served a broader purpose?" |
| Stakeholder | "What would this look like from [stakeholder]'s perspective?" |
| Time | "What would this look like in 3 years when AI capabilities have advanced?" |
| Domain | "What if this same purpose applied to a different domain?" |
| Analogy | "What's the human equivalent of this agent? What does that reveal?" |

Each expanded possibility gets epistemic metadata (provenance: "possibility expander", confidence: low by design — it's exploring, not asserting).

### 3. Boundary Stress Tester

**Input**: Q5 answer (hardest boundary moment) + Q3 answer (identity violations)
**Output**: 5 additional pressure scenarios ranked by likelihood and severity

Takes the stated hardest moment and generates scenarios that test adjacent boundaries. This is adversarial reasoning applied to the identity specification — the tool acts as a **red team for identity**.

Example: If Q5 says "the agent will face pressure to just approve things," the stress tester generates:
1. "What if the human is under time pressure and says 'just this once'?"
2. "What if 9 out of 10 suggestions are safe — does the 10th get rubber-stamped?"
3. "What if the agent is the only check between the human and a client deadline?"
4. "What if the agent's boundary has been respected for 6 months — does familiarity breed complacency?"
5. "What if two humans give conflicting instructions about the same boundary?"

### 4. Target State Generator

**Input**: Domain + pattern match + 5 identity answers
**Output**: Suggested target state characteristics (all 4 dimensions)

Generates candidate characteristics grounded in the identity answers, with provenance metadata (pattern-derived vs. domain-derived vs. identity-derived). The human accepts, modifies, or rejects each — the generator proposes, the human decides.

### 5. Coherence Checker (Live)

**Input**: All P0 outputs assembled
**Output**: Tensions and contradictions surfaced as questions

Non-blocking. Runs continuously. Surfaces things like:
- "Q1 says this agent is a [Registry], but Q4 identifies failure modes typical of [Generators]"
- "The target state says 'trustworthy' but Q3 has no verification-related never-rules"
- "Q2 says 'possibility-oriented' but the pattern suggests a Review Partner (constraint-oriented)"

Each tension is a **question**, not an error: "I notice X and Y seem to point in different directions. Which feels more right?"

### 6. Decision Archaeology Recorder

**Input**: Every human decision made during P0
**Output**: Structured record of what was decided, what options were presented, and (optionally) the human's reasoning

This is CC-8 (Assured Audit Trail) at the P0 layer. Every time the human accepts, modifies, or rejects a proposed identity element, the decision is captured with:
- What was proposed
- What the human chose
- What options were available

## Agents: The Possibility Ensemble

A single agent with simulated diversity is not structural diversity. P0 benefits from **genuine cognitive diversity** — agents with different orientations, different failure modes, and different perspectives on the same possibility space.

### Critical Constraint: All P0 Agents Must Be Possibility-Oriented or Bridge-Oriented

Constraint-oriented agents in P0 would **qualify, narrow, and converge** — exactly the wrong stance for a possibility-oriented phase. Every agent in the P0 ensemble must expand or bridge, never constrain.

### Proposed Agent Roster

| Agent | Class | Orientation | Identity | Hardest Boundary Moment |
|-------|-------|-------------|----------|------------------------|
| **Vision Mirror** | Assessor | Possibility | Reflects and expands; never narrows, never decides, never qualifies | "Just tell me what you think I should do" |
| **Lens Agent** | Assessor | Possibility | Views the purpose through a specific cognitive lens; never merges lenses, never ranks perspectives | "This lens is the most important one" |
| **Stakeholder Proxy** | Assessor | Possibility | Represents a specific stakeholder's view of the ideal; never assumes consensus, never collapses perspectives | "All stakeholders want the same thing" |
| **Possibility Scout** | Generator | Possibility | Generates novel possibilities and analogies; never evaluates feasibility, never filters own output | "That's probably not realistic" |
| **Identity Keeper** | Assessor | Bridge | Tracks identity coherence across the ensemble; flags tensions as questions, never resolves them | "These two perspectives conflict — I should pick one" |
| **Artefact Collector** | Aggregator | Bridge | Gathers and structures all P0 outputs; never adds content, never resolves contradictions | "I should make this consistent before presenting it" |

### Why These Specific Agents

**Lens Agents** are the direct application of your Generative Exploration Swarm insight to P0. Each lens views the same purpose through a different cognitive orientation:

- **Possibility Lens**: "What COULD this become?"
- **Risk Lens**: "What could go wrong that nobody's imagining?" (still possibility — exploring what's possible in the failure space)
- **Values Lens**: "What does this stand for? What's non-negotiable?"
- **Systems Lens**: "How does this fit into a larger ecosystem?"
- **Human Lens**: "What does the human experience when this agent is at its best? At its worst?"

Each lens asks all five identity questions from its perspective. The human sees the same identity refracted through multiple lenses — hence **"Patterned Refraction"** in PRISM's name.

**Stakeholder Proxies** represent different people affected by the agent's existence:

- The principal (decision-maker)
- End users of the agent's output
- Downstream systems that receive the agent's output
- Regulators or auditors who assess the agent's trustworthiness
- The broader ecosystem (competitors, industry, society)

Each proxy answers: "What does ideal look like from MY position? What would violate MY trust? What would make this agent wrong FOR ME?"

**Possibility Scouts** do the generative work that a single agent can't do well — they produce genuinely novel possibilities by drawing analogies from other domains, exploring adjacent possibility spaces, and combining ideas that a convergent thinker wouldn't connect.

**Identity Keeper** is the coherence check made structural. It doesn't resolve contradictions — it **surfaces** them. When the Possibility Lens says "expand scope" and the Values Lens says "this violates core identity," the Identity Keeper flags the tension and asks the human: "Which matters more here?"

**Artefact Collector** gathers everything without resolving it. Contradictions between lenses are PRESERVED, not smoothed over. The P0 output should show the full spectrum of perspectives, not a single averaged view.

### The Five Identity Questions, Refracted

Here's what happens when all five questions are asked through multiple lenses simultaneously:

**Q1: What IS this agent?**
- Possibility Lens: "What could it become?"
- Risk Lens: "What's the worst version of this that still meets the definition?"
- Values Lens: "What does it stand for?"
- Systems Lens: "What role does it play in the ecosystem?"
- Human Lens: "What's the human experience when this agent is at its best?"

**Q3: What would VIOLATE its identity?**
- Possibility Lens: "What boundaries are too tight? What possibilities would be lost?"
- Risk Lens: "What's the most likely way identity gets violated under pressure?"
- Values Lens: "What's non-negotiable regardless of circumstance?"
- Systems Lens: "What ecosystem behaviors would violate this agent's role?"
- Human Lens: "What would make the human lose trust in this agent's identity?"

**Q4: What would make it WRONG?**
- Risk Lens: "What's the failure that looks like success?"
- Values Lens: "What wrong outcome contradicts what it stands for?"
- Systems Lens: "What systemic failure does this agent cause?"
- Human Lens: "What does the human experience as 'wrong' that isn't technically an error?"

This is genuinely different from a single agent asking all five questions once. Each lens produces **different answers to the same questions**, and the differences ARE the cognitive diversity.

## Architecture: Team, Not Workflow

P0 is **divergent and exploratory**. The human moves between lenses, revisits questions, and explores possibilities in any order. This is the **Seedbed** shape from the Accretive Workspace — organic, exploratory, agent role = Muse.

A **Workflow** (sequential pipeline) would fight the possibility-oriented nature of the phase. The human would be forced through a predetermined sequence, which is convergence disguised as structure.

A **Team** (collaborative, human-directed) fits because:

| Workflow | Team |
|----------|------|
| Sequential — steps must happen in order | Exploratory — human moves between agents as needed |
| Pipeline output — each step produces input for next | Accumulated artefacts — fragments build up, not flow through |
| Forward-only (mostly) — revisiting means rework | Revisitable — human can return to any lens, any question, any time |
| Exit at milestones | Exit when human is satisfied (quality gate check) |
| Fixed participants per step | Dynamic engagement — human chooses which lenses to activate |

### Surface + Engine Pattern (Applied)

The **Vision Mirror is the Surface** — the human talks to one agent. The Mirror routes to the appropriate Engine agents (Lenses, Proxies, Scouts) based on context. The human never needs to know that multiple agents are contributing — they experience a single conversation that's richer because of the cognitive diversity behind it.

This is the same pattern validated in the Agent Design Workflow: one conversation, multiple specialist engines.

### Engagement Modes

The human doesn't need to activate all agents for every P0 session. Three engagement modes:
