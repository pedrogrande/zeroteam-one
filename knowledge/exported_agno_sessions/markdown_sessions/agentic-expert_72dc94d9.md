# Design Space: Capturing and Retrieving Session Knowledge

---

## The Design Challenge

Capture session outputs as typed, connected fragments — without extra human work — so the agent can retrieve the right knowledge at the right time.

---

## Three Decisions, Not One

This isn't one problem. It's three separate problems that interact:

**1. Capture** — How does knowledge get INTO the system?

**2. Structure** — What shape does it take once it's IN?

**3. Retrieval** — How does knowledge get OUT when needed?

These aren't independent. The capture method determines what structure is possible. The structure determines what retrieval is possible. And the retrieval need determines what capture is worth doing.

But they can be designed for independently if we know the constraints of each.

---

## Capture — How Knowledge Gets IN

### The Constraint: No Extra Human Work

Peter should not have to:
- Manually tag outputs during a session
- Fill in forms after a session
- Review and categorise every fragment
- Maintain a knowledge base

Any solution that requires these fails the constraint.

### The Options

**Option A: Post-Session Extraction**

After each session, a dedicated process (an agent, a script, or a combination) processes the full transcript and extracts typed fragments.

- **What Peter does:** Nothing. The system runs automatically.
- **What the system does:** Reads the transcript, identifies outputs, types them, connects them to existing fragments, stores them.
- **Accuracy:** Moderate. An extraction agent will miss subtleties. A reframing might be classified as a new concept. A connection might not be made.
- **Effort for Peter:** Zero. But he might not trust what he didn't review.

**Option B: In-Session Markers**

During the session, the agent notes "output produced" markers — lightweight flags that say "something important just happened here" without stopping the conversation.

- **What Peter does:** Nothing. The markers are automatic.
- **What the system does:** Places markers at significant moments. A post-session process uses the markers to focus extraction on high-value segments.
- **Accuracy:** Higher than Option A, because the extraction is guided by markers rather than scanning an undifferentiated transcript.
- **Effort for Peter:** Nothing during the session. Possibly a quick review of extracted fragments.

**Option C: In-Session Capture + Immediate Storage**

During the session, the agent both produces the output AND creates a typed fragment in the same turn. "We just discovered that type collision is bidirectional — this is a reframing of the original concept."

- **What Peter does:** Nothing, unless the typing is wrong and he wants to correct it.
- **What the system does:** Produces typed fragments as part of the response, stores them immediately.
- **Accuracy:** Highest, because the typing happens at the moment of creation when context is richest.
- **Effort for Peter:** Nothing, but the agent's responses might feel slightly more structured.

**Option D: Hybrid — Markers During, Extraction After, Review Optional**

During the session, the agent marks significant moments. After the session, a process extracts and types. Peter can review high-stakes items (principles, decisions, corrections) but doesn't have to.

- **What Peter does:** Optionally review high-stakes extractions. Can skip entirely.
- **What the system does:** Marks during session, extracts after, types, connects, stores. Flags items where confidence is low for optional review.
- **Accuracy:** Good. Extraction is guided by markers. Low-confidence items can be reviewed or left.
- **Effort for Peter:** Zero to minimal. Review is optional, not required.

| Option | Peter Effort | Accuracy | Timeliness | Complexity |
|--------|-------------|----------|------------|------------|
| A: Post-Session | Zero | Moderate | Delayed | Low |
| B: In-Session Markers | Zero | Good | Delayed | Medium |
| C: In-Session Capture | Zero | Highest | Immediate | Higher |
| D: Hybrid | Zero-Minimal | Good-High | Mixed | Medium |

Option C is the most interesting because it's closest to what Peter already designed for the Accretive Workspace — capture fragments as they're produced, type them, connect them. It's the product eating its own dog food.

Option D is the pragmatic choice — markers during the session, extraction after, optional review. But it depends on a good extraction process.

- A specification is obviously a specification
- A definition is obviously a definition

- Is this a principle or a learning?
- Is this a connection that's always been there, or one we just discovered?

The accuracy of real-time typing depends on how well the agent can distinguish these categories. Some are unambiguous. Others require judgment that might be better done in review.

## Retrieval — How Knowledge Gets OUT

### The Constraint: Relevant and Fast

The agent needs the right fragments at the right time. Not everything. Not a wall of text. The right knowledge, structured for the current need.

### The Retrieval Options

**Option 1: Pre-Session Injection**

At the start of each session, inject relevant fragments based on the session topic.

- **How it works:** Peter states the session topic. The system retrieves all fragments related to that topic, ranked by recency and confidence.
- **Strength:** Peter starts each session with relevant context.
- **Weakness:** What if the session topic changes mid-session? What if the most relevant fragment is about a different subject?

**Option 2: On-Demand Query During Session**

During the session, the agent queries the knowledge base when it needs specific information.

- **How it works:** When Peter asks about PRISM, the agent queries all PRISM fragments. When Peter mentions type collision, the agent queries all type collision fragments.
- **Strength:** Precise, just-in-time retrieval.
- **Weakness:** The agent might not know when it needs to query. It might not know what it doesn't know.

**Option 3: Pre-Session Summary + On-Demand Query**

Combine both: a structured summary at session start (for orientation) + on-demand queries during the session (for precision).

- **How it works:** Start with a summary of what's most relevant to the stated topic. During the session, query for specific fragments when precision is needed.
- **Strength:** Both breadth (summary) and depth (queries).
- **Weakness:** More complex system to build.

**Option 4: Continuous Background Retrieval**

The system continuously monitors the conversation and retrieves relevant fragments in the background, injecting them when they become relevant.

- **How it works:** As the conversation develops, the system identifies subjects being discussed and proactively retrieves related fragments.
- **Strength:** No manual querying needed. The system anticipates needs.
- **Weakness:** Might retrieve too much (noise) or too little (missing critical context).

### What I Notice About These Options

Option 3 is the most interesting because it matches Peter's own ACP design — SELECT (budget-aware, relevance-scored, confidence-weighted context windowing) for initial retrieval, then ENRICH (7-dimension traversal) for precision, then CALIBRATE (attach epistemic metadata) for confidence.

The question underneath all these options: **What triggers retrieval, and how much is too much?**

The context ceiling we started with is exactly this problem — too much context degrades quality, too little misses connections. The retrieval system needs to be surgical: give me exactly what I need, not everything I might need.

This is where type-based retrieval becomes critical. If I can say "give me all current principles about CAWDP, ranked by recency," that's surgical. If I can only say "give me everything about CAWDP," that's a wall of text.

## How Might a Curator Agent Judge What to Persist?

### The Judging Problem

Not everything said in a session is an output. Most of a session is exploration, questions, false starts, refinement. The curator needs to distinguish:

- "What if type collision applies to inputs too?" → This is a **question** being explored, not yet an output
- "Type collision IS bidirectional" → This is a **reframing** — a genuine output
- "Let me think about that" → This is process, not output
- "The boundary IS the product" → This is a **naming moment** — high-value output

The difference between exploration and output is often clear in the moment but hard to detect after the fact.

### Judging Mechanisms

**1. Signal Detection**

The curator watches for signals that an output has been produced:
- Naming language: "I'm going to call this...", "This is...", "The key insight is..."
- Certainty language: "IS" rather than "might be", "we've established" rather than "let's explore"
- Reframing language: "Actually, it's not X, it's Y", "What we've been calling X is really Y"
- Decision language: "We're going with...", "The decision is...", "From now on..."
- Rejection language: "That approach doesn't work because...", "We rejected X because Y"

These signals indicate that something has moved from exploration to output.

**2. Confidence Scoring**

Each detected output gets a confidence score based on:
- **Source confidence**: Was this stated explicitly by Peter, derived from Peter's direction, or generated by the agent?
- **Stability confidence**: Has this been stated once or repeatedly? Does it match previous outputs or contradict them?
- **Type confidence**: Is the output type clear (this is definitely a principle) or ambiguous (could be a learning or a hunch)?
- **Subject confidence**: Is the subject clear (definitely about CAWDP) or fuzzy (related to multiple subjects)?

A high-confidence output (explicitly stated, repeated, clear type, clear subject) is more likely to be a genuine output worth persisting. A low-confidence output might be exploration worth flagging but not formalising.

**3. Novelty Detection**

Before storing, the curator checks: does this already exist in the knowledge base?
- **Exact duplicate**: Don't store again. Update recency if it's a restatement.
- **Supersession**: This output replaces an earlier one. Mark the earlier one as superseded, store the new one with a "supersedes" link.
- **Refinement**: This output refines an earlier one. Link it as a refinement, update the original's recency.
- **Contradiction**: This output contradicts an earlier one. Flag for review — is this a correction or a new perspective?
- **New**: This output has no existing match. Store as new.

**4. Value Estimation**

Not all outputs are equally worth persisting. A principle that will guide future work is more valuable than a fleeting observation. Value could be estimated based on:
- **Output type**: Principles, decisions, and corrections tend to be higher-value than ideas and questions.
- **Reusability**: Does this apply to one specific thing or to many things?
- **Stability**: Has this been stable across sessions, or is it the first time it's been stated?
- **Connections**: Does this connect to multiple existing outputs, suggesting it's a hub idea?

**5. Epistemic Metadata as the Judging Layer**

You're right that this is where epistemic metadata becomes very useful. Each potential fragment gets:
- **Confidence**: How certain is the curator that this is a genuine output?
- **Provenance**: Where did this come from? Explicit statement by Peter? Derived from Peter's direction? Generated by the agent?
- **Assumptions**: What assumptions did the curator make in extracting this?
- **Limitations**: What might be wrong or incomplete about this extraction?
- **Recency**: When was this produced? Is it the latest version?
- **Uncertainty**: What doesn't the curator know about this output?

These six fields (the same CC-3 epistemic metadata from CAWDP) serve a dual purpose: they help the curator judge whether to persist, and they help future retrieval judge whether to trust what was persisted.

Agno gives us capture, storage, and basic retrieval. But some things we'd need to build on top:

**Fragment Relationships**

Agno's stores are per-type, not a connected graph. We'd need to build:
- Supersession tracking (this replaces that)
- Connection typing (this extends that, this contradicts that)

**Confidence-Based Retrieval**

Agno doesn't natively score retrieval results by confidence. We'd need to:
- Store confidence in the custom schema
- Build retrieval logic that filters by confidence threshold
- Update confidence as outputs are validated or superseded

**Temporal Queries**

"Give me the current version of this concept" requires knowing which version is current, which is superseded. Agno doesn't have built-in versioning for this.

**Cross-Type Queries**

"Give me all outputs related to CAWDP, regardless of type" requires querying across all stores with a subject filter. This might be possible with Agno's existing infrastructure but would need custom query logic.
