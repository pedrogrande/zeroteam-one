
# Task Contract — Master Outline

***

## Preamble

- Contract identifier and version
- Contract creation timestamp
- Governing protocol and ruleset version
- Jurisdiction (legal, organisational, or decentralised protocol)
- Contract type (one-off, recurring, ad hoc)
- Contract status

***

## I. Origin & Intent

### 1.1 Originating Desire

- Human desire statement
- Problem or gap being addressed
- Current state description
- Desired future state description

### 1.2 Strategic Linkage

- Parent desire / goal / OKR / initiative
- Value stream position
- Outcome contribution statement
- Entropy resolved (uncertainty this task eliminates)

### 1.3 Creation Provenance

- Created by (actor ID)
- Creation source (human-initiated, process-triggered, agent-generated, event-triggered)
- Creation timestamp
- Originating context reference (meeting, incident, idea, obligation)

***

## II. Task Specification

### 2.1 Identity

- Task title
- Task description (full narrative)
- Task type / work nature classification
- Functional domain
- Value stream category
- Tags and labels

### 2.2 Scope

- Explicit inclusions
- Explicit exclusions
- Assumptions
- Constraints

### 2.3 Definition of Done

- Acceptance criteria (enumerated, each independently verifiable)
- Verification method per criterion
- Minimum quality threshold
- Target quality standard

### 2.4 Expected Outputs

- Artefact list (type, format, destination)
- State changes to be effected
- Decisions to be made and recorded
- Communications to be sent
- Handoff specification (recipient, format, timing)

### 2.5 Complexity & Effort

- Complexity rating
- Cognitive load type
- Estimated effort (minutes)
- Estimated lead time (minutes)
- Maximum permitted duration (the 4-hour ceiling)
- Flow potential rating
- Focus mode requirement

***

## III. Value Declaration

### 3.1 Value to Requester

- Value type (financial, operational, strategic, relational, informational)
- Estimated value quantum
- Value realisation timeframe
- Value decay profile (none, linear, exponential, cliff-edge)

### 3.2 Value to Performer

- Compensation (token, monetary, credit, reputational)
- Skill development opportunity
- Knowledge gain
- Portfolio / reputation contribution

### 3.3 Value to Organisation

- Strategic alignment score
- Contribution to mission
- Leverage score (systemic impact relative to effort)
- Waste classification (value-creating, enabling, protecting, discovering)

### 3.4 Value to Broader Stakeholders

- Third-party beneficiaries
- Community or societal value
- Negative externalities declared

### 3.5 Value Verification

- How value realisation will be assessed post-completion
- Value realisation timeline
- Accountable assessor

***

## IV. Parties & Roles

### 4.1 Requester

- Requester ID
- Requester type (human, organisation, agent, protocol)
- Authority to issue this contract
- Desire statement in requester's own terms

### 4.2 Performer

- Performer ID (assigned upon claiming)
- Performer type (human, agent, hybrid)
- Claim timestamp
- Claim mechanism (assigned, self-selected, auctioned, delegated)
- Delegability permissions

### 4.3 Owner / Accountable Party

- Owner ID
- Distinction from performer if different
- Escalation authority

### 4.4 Reviewer(s)

- Reviewer ID(s)
- Review type (quality, compliance, safety, domain expertise)
- Review authority level
- Sequential or parallel review structure

### 4.5 Verifiers

- Verifier ID(s) per acceptance criterion
- Verification independence requirement
- Decentralised verification quorum (if applicable)
- Verification method per artefact type

### 4.6 Stakeholders & Observers

- Stakeholder ID
- Role (impacted, sponsor, beneficiary, regulator)
- Notification entitlements
- Input rights (none, advisory, approval)

***

## V. Capability Requirements

### 5.1 Skills

- Required skill(s) and proficiency level
- Preferred skill(s)
- Assessment method for skill verification
- Skill ruleset version referenced

### 5.2 Knowledge Domains

- Required knowledge domains
- Organisation-specific context required
- Onboarding or briefing materials referenced

### 5.3 Tools & Access

- Required tools and platforms
- Access credentials or permissions needed
- Licensing requirements

### 5.4 Experience Level

- Minimum experience classification
- Experience ruleset version referenced

### 5.5 Performer Eligibility

- Eligibility criteria
- Disqualifying conditions
- Trust score minimum (if applicable)
- Reputation threshold (if applicable)

***

## VI. Inputs & Dependencies

### 6.1 Required Inputs

- Input ID and description
- Input type (data, decision, artefact, access, briefing)
- Provided by (actor ID)
- Required before start flag
- Input availability status

### 6.2 Task Dependencies

- Depends on (task contract IDs)
- Blocks (task contract IDs)
- Dependency type (hard, soft, informational)
- Dependency resolution status

### 6.3 Earliest Start Condition

- Earliest start date/time
- Trigger condition for availability
- Pre-conditions checklist

***

## VII. Scheduling & Timing

### 7.1 Temporal Bounds

- Earliest start date
- Due date
- Deadline type (hard, soft)
- Expiry condition (what happens if not claimed or completed by deadline)

### 7.2 Recurrence

- Recurrence type (once, scheduled, ad hoc)
- Recurrence schedule (cron expression)
- Instance number (for recurring tasks)
- Parent recurrence contract reference

### 7.3 Time Sensitivity

- Time sensitivity classification
- Time sensitivity ruleset version referenced

***

## VIII. Priority & Allocation

### 8.1 Priority

- Priority classification (must-have through won't-have)
- Priority score (computed)
- Priority ruleset version referenced
- Priority last reviewed timestamp

### 8.2 Allocation Mechanism

- Allocation type (assigned, claimable, auctioned, delegated, automated)
- Claim window (open from / open until)
- Maximum concurrent claims permitted
- Allocation authority (if assigned)

***

## IX. Risk & Safety

### 9.1 Risk Register

- Risk ID, type, description
- Likelihood rating
- Consequence severity
- Risk tier (1–4)
- Mitigation strategy
- Risk owner

### 9.2 Risk Types Declared

- Non-completion risk
- Execution / quality risk
- Safety risk (physical, psychological, legal, reputational)
- Dependency risk
- Data / security risk
- Compliance / regulatory risk
- Reputational risk

### 9.3 Performer Safety

- Safety obligations on requester
- Safety obligations on performer
- Escalation path for safety concerns
- Performer safety ruleset version referenced

### 9.4 Blast Radius & Reversibility

- Blast radius classification
- Reversibility classification
- Required safeguards before execution
- Approval requirement tier

***

## X. Quality

### 10.1 Quality Standards

- Target quality level
- Quality criteria per output
- Quality ruleset version referenced

### 10.2 Quality Assessment

- Assessed by (reviewer ID)
- Assessment method
- Achieved quality (recorded post-completion)
- Quality variance from target

***

## XI. Confidentiality & Governance

### 11.1 Confidentiality

- Confidentiality classification
- Handling obligations
- Permitted actors (who may view this contract and its outputs)
- Data retention requirements
- Confidentiality ruleset version referenced

### 11.2 Compliance Obligations

- Regulatory or policy obligations this task must satisfy
- Compliance verification method
- Compliance verifier ID

### 11.3 Governing Rulesets

- Full enumeration of all rulesets referenced, with version numbers

***

## XII. Compensation & Incentives

### 12.1 Base Compensation

- Compensation type (monetary, token, credit, reputational, in-kind)
- Compensation amount or formula
- Payment trigger (on completion, on verification, on value realisation)
- Payment mechanism

### 12.2 Quality Incentives

- Bonus conditions (e.g. exceeds target quality, early completion)
- Penalty conditions (e.g. rework required, deadline missed)
- Incentive ruleset version referenced

### 12.3 Reputation Effects

- Reputation impact on successful completion
- Reputation impact on failure or cancellation
- Reputation system referenced

***

## XIII. Execution & State

### 13.1 Lifecycle States

- Valid states enumerated
- Permitted state transitions
- State transition trigger conditions
- State transition authority (who can move between states)

### 13.2 Pause & Cancellation

- Conditions under which task may be paused
- Conditions under which task may be cancelled
- Consequences of cancellation (compensation, reputation, downstream effects)
- Cancellation authority

### 13.3 Dispute Resolution

- Dispute trigger conditions
- Resolution mechanism (human arbitration, protocol rule, DAO vote)
- Dispute resolution timeline
- Escalation path

***

## XIV. Verification & Proof of Completion

### 14.1 Completion Proofs

- Proof type per acceptance criterion
- Proof submission method
- Proof storage location (on-chain reference, IPFS hash, document store ID)
- Proof immutability requirement

### 14.2 Verification Process

- Verification sequence (which criteria verified in what order)
- Independent verification requirement
- Decentralised verification quorum specification
- Verification deadline

### 14.3 Artefact Registry

- Artefact ID, type, format
- Submission timestamp
- Verifier ID and verification timestamp
- Pass/fail status per artefact
- Immutable storage reference

***

## XV. Notifications & Coordination Signals

### 15.1 Notification Rules

- Trigger events for notifications
- Recipients per trigger
- Notification channel
- Notification ruleset version referenced

### 15.2 Coordination Signals

- Signal type emitted on each state transition
- Downstream systems or actors to be notified
- Bottleneck flag conditions

***

## XVI. Learning & Knowledge

### 16.1 Experiment Record

- Expected outcome (stated before execution)
- Actual outcome (recorded post-completion)
- Outcome variance
- Task validated (retrospective assessment)

### 16.2 Retrospective

- Retrospective trigger conditions
- Retrospective format reference
- Retrospective notes
- Retrospective conducted by
- Retrospective timestamp

### 16.3 Knowledge Output

- Knowledge output flag
- Output type (template, decision record, process documentation, insight)
- Knowledge base reference
- Reusability classification
- Knowledge building guidelines version referenced

### 16.4 Capability Development

- Skill development opportunity flag
- Supervising actor (if stretch assignment)
- Competency evidence generated

***

## XVII. Audit & Immutable Record

### 17.1 Audit Trail

- Full log of all state transitions (actor, timestamp, reason)
- Full log of all edits to contract terms (before/after, actor, timestamp)
- Full log of all assignment and claim events
- Full log of all verification events

### 17.2 Immutability

- Immutable record protocol (blockchain, hash chain, signed log)
- Record storage reference
- Hash of final contract state at completion
- Witness / notarisation (if applicable)

### 17.3 Historical Lineage

- Parent task contract reference (if decomposed from a larger task)
- Child task contract references (if this task was decomposed)
- Prior version references (if contract was renegotiated)

***

## XVIII. Signatures & Commitments

### 18.1 Requester Commitment

- Requester signature / cryptographic key
- Commitment timestamp
- Declaration of mutual understanding

### 18.2 Performer Commitment

- Performer signature / cryptographic key
- Commitment timestamp
- Acknowledgement of DoD, capability requirements, and terms

### 18.3 Reviewer / Verifier Acceptance

- Reviewer signature(s)
- Acceptance of verification obligations

### 18.4 Witnessing

- Witness ID (human, agent, or protocol)
- Witness timestamp
- Witness type (organisational, decentralised, notarial)
