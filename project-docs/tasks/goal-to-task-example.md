Goal - tasks

We're into the specified work stage now. Here's what it does, then the decomposition.

**What specification does:** it takes the gap and asks *what set of tasks actually closes it* — then types each task, inherits value and lineage down from the intent, and surfaces the unknowns the "deliver" framing hid.

**Why backcasting fits here:** you already have the destination (`desired_state`) but not the path. So you start at the goal and step backwards, asking at each point *"what must be true immediately before this?"* Its real virtue for you: every task gets **derived from the goal**, so no orphan work can appear. It structurally enforces your "no activity that doesn't serve the goal" principle.

## The backcast — working backwards

- **Goal:** 10 contributors active, each with one *verified* first task.
- **Immediately before:** people are onboarded **and** starter tasks are waiting **and** verification works. *(Three things must be true at once — hold that, it matters.)*
- **Before that:** people were invited, said yes, and the experience exists to receive them.
- **Before that:** we know *who* we want and *what* we're offering them.

Flipped to read forwards, that's the plan:## The tasks inside each phase

**Foundation**

- **T1 — Define the founding-contributor profile.** *(discover — you have a guess, you'll refine it)*
- **T2 — Define the offer + terms.** What they get, what's expected, how contribution is recognised. *(deliver)*

**Build stream**

- **T3 — Design the onboarding flow.** *(deliver, with discover inside — you don't yet know what works)*
- **T4 — Create real starter tasks** people can claim, each mapped to the task schema. *(deliver)*
- **T5 — Set up first-task verification.** *(deliver)*

**People stream**

- **T6 — Build a candidate list** *(deliver — needs more than 10, since not everyone says yes)*
- **T7 — Write the invitation / pitch.** *(deliver — depends on T1 and T2)*
- **T8 — Invite candidates.** *(recurring per candidate)*
- **T9 — Onboard those who accept.** *(recurring per person)*

**Convergence**

- **T10 — Support each person to claim + complete a first task.** *(recurring per person)*
- **T11 — Verify each first task.** *(recurring per person)*

## The insight backcasting hands you

Look at the convergence point: **three things must be true at once** — onboarded people, starter tasks, and working verification. That forces a sequencing rule the forward view hides:

> **Infrastructure before invitation.** Don't invite founders until the experience can receive them.

Your founding cohort is your scarcest, least-replaceable resource. Invite them into a broken first experience and you burn the exact people you can't get back. The exciting move (start inviting now) is the wrong one. Build T3–T5 first.

And a recursive option that's very *you*: **cohort one's first tasks could be building the experience for cohort two.** But that needs a tiny hand-cranked bootstrap — a few starter tasks made by hand, verification done manually for the first few — before it can scale. Worth noting, not solving now.

## One task, mapped to the schema

To show the inheritance works — here's **T1**, the discover-task at the root:

- **change:** current = no profile of who we want; desired = clear, testable founding-contributor profile
- **intent_kind:** discover
- **floor / target / direction:** floor = a usable working profile; target = profile validated against 3+ real candidates; direction = sharper fit over time
- **value_vector (inherited + local):** knowledge (high — everything downstream uses it), capability (medium)
- **goal_logic:** terminating
- **parent:** the onboarding intent above it — so it inherits the mission lineage automatically

Most of T1's fields **flowed down** from the parent intent. Only the local specifics were authored. That's the efficiency principle working: cheap to specify because it inherits.

## What this confirms

Last turn you flagged the parent intent as *"deliver-type with discover lurking inside."* The decomposition **proves it** — T1 and T3 came out as discover-tasks nested inside a deliver-intent. The prediction held. That's the loop closing on its own forecast, exactly the epistemic behaviour you designed in.

One watch-point: the **floor (5) vs target (10)** spread barely changes *these* tasks — the same infrastructure serves 5 or 10. So here the band is informative for *payout*, not *decomposition*. Good to know the two don't always couple.
